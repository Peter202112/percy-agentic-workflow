Quick Start

# 1. Open the Percy monorepo in Claude Code
cd Percy
claude

# 2. Start from a problem statement (gated workflow)
@percy-orchestrator "Percy users spend too long reviewing visual
diffs manually. They have no way to prioritize which diffs matter most."

# 3. Or start from a PRD
@percy-orchestrator Here's the PRD: [paste or reference file]

# 4. Run QA independently
/percy-qa

# 5. Start local dev server
/percy-local

The gated orchestrator guides you through 8 stages with 9 approval gates. Every stage pauses for your explicit approval — feedback loops back to re-run that stage. You can also invoke any agent individually — see Using Individual Agents.

Workflow Stages

Each stage produces an HTML report in reviews/ and pauses at an approval gate. Feedback re-runs the current stage — no downstream work is wasted.

1

Research

percy-researcher + code-explorer

Competitive analysis, codebase mapping, 3 solution directions. Includes User Benefit Validation: problem solved, impact metric, alignment check.

reviews/research-{feature}.html
🚦 Gate: Approve research findings + alignment
2

Product Brief

percy-pm

Personas, phased deliverables, acceptance criteria, edge cases, success metrics.

reviews/product-brief-{feature}.html
🚦 Gate: Approve scope + criteria
3

Design MANDATORY

percy-design-explorer + Figma MCP

Blocks until user provides: (1) Figma page URL for mocks, (2) design system reference URL. Extracts tokens from reference, creates 3+ screens. After approval, mockups are embedded into the product brief (single source of truth).

Figma mockups + reviews/design-exploration-{feature}.html + updated product brief
🚦 Gate: Approve mockup screenshots. Explicit "skip Figma" required to bypass.
4

Implementation Plan

code-architect

DB schema, API models/services/controllers, Web models/components, file-by-file change list, data flow diagram. Uses updated product brief with embedded mockups.

Architecture blueprint
🚦 Gate: Approve architecture + file list
5

Implementation (API + Web)

Direct code writing

API: migration → models → services → job → serializer → controller → routes. Web: model → adapter → service → components → route. ESLint auto-fix.

🚦 Gate 5a: Approve API diff  |  🚦 Gate 5b: Approve Web diff
6

QA Verification

/percy-qa

Docker setup, API service specs, API request specs, frontend build, ESLint. Produces comprehensive test report with detailed test plan, suite, and cases (objective, steps, data, expected/actual, notes).

reviews/testresults.md + testresults.html
🚦 Gate: Approve test results
7

Commit & PR

git + gh pr create

Commit in both repos with descriptive messages. Push and create PR only after approval.

🚦 Gate: Approve push to remote
8

Cleanup

Infrastructure shutdown

Presents running resources (Docker, dev server, browser). Warns that restarting Docker takes 2-5 min. Three options: shut down all, keep Docker (recommended), keep everything. Default: keep everything running.

🚦 Gate: Explicit approval required to shut down (default: keep running)

Why Agents Live Where They Do

Claude Code scopes agent discovery by working directory. When you cd percy-web && claude, you see percy-web agents + root agents, but NOT percy-api agents (and vice versa). This keeps each context clean and relevant.

LocationAvailable WhenWhat Goes Here
.claude/ (root) Always — from any directory Workflow agents (orchestrator, researcher, PM, design-explorer), cross-repo agents (contract reviewer, usability reviewer), shared skills
percy-web/.claude/ Only in percy-web Web-specific reviewers (design-system, component, template, ember-data, accessibility), web commands, web PR validation
percy-api/.claude/ Only in percy-api API-specific reviewers (endpoint, job, spec, security, migration, etc.), API commands, API PR validation

Rule of thumb: If an agent reads/writes code in only one repo, it lives in that repo's .claude/. If it spans both repos or operates at the product level (research, design, usability), it lives at the root.

Review Reports

All HTML reports are self-contained (inline CSS, no external dependencies). Open in any browser. The product brief is the single source of truth — it includes embedded mockup screenshots after design approval.

reviews/
├── research-{feature}.html              # Stage 1: Competitive analysis + user benefit validation
├── product-brief-{feature}.html         # Stage 2+3: Brief + embedded Figma mockup screenshots
├── design-exploration-{feature}.html    # Stage 3: 3 approaches, comparison matrix
├── testresults.md                       # Stage 6: Detailed test plan, suites, cases (markdown)
├── testresults.html                     # Stage 6: Styled test report (HTML)
├── validation-{feature}.html            # Stage 6: Validation summary
├── screenshot-*.png                     # Stage 5: Browser automation screenshots
└── test-results-{feature}.html          # Stage 6: Full pass/fail with spec output

Suppression

All review agents support percy:ignore {rule-id} - {reason}. Suppressed findings are tracked in reports, not hidden.

Ruby (percy-api)
# percy:ignore endpoint-safety/unguarded-skip-authorization - superuser verified
Handlebars (percy-web)
{{!-- percy:ignore design-system/hardcoded-color - legacy pending migration --}}
JavaScript (percy-web)
// percy:ignore component/magic-number - timeout from API spec
